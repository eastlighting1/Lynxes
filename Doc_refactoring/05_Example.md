# 🧨 [Examples] 예제 문서 리팩토링 지시서 (초강도/상세본)

**"작동 자체가 불가능한 죽은 예제 코드는 개발자를 배신한다. 철저하게 CI가 빌드를 감시하는 100% 검증 코드만 임베드하라."**

현재 `docs/gf_authoring_guide.md` 등 제품 안내서들 곳곳에 마크다운 블록 렌더링으로 어쭙잖게 직접 사람의 손으로 직접 타이핑해 끼워넣은 파편적 코드스니펫들은 가장 질이 나쁜 형태의 기술 부채(Technical Debt)입니다. 제품의 코어 API 스펙이 한 개라도 업데이트되는 그 순간, 문서상의 글자들은 업데이트를 따라가지 못한 채 버려지고, 이를 믿은 수많은 유저들의 코드를 망치는 주범이 됩니다.

## 🚨 1. 현재 문서 생태계의 치명적 결함 및 해부 구조

**결함 1: `gf_authoring_guide.md`의 문법 변화로 인한 방치 폭탄 위험성**
*   가장 뚜렷한 예로 현재 파일의 135번부터 163번 라인 사이에 길게 포진한 거대한 `.gf` 스타일 작성 가이드 예제가 단순히 수기 마크다운 텍스트 덩어리로 박혀 있습니다. 만천하에 공개된 이 문서상태에서, 당장 내일 메인 엔진 코어 코드에서 `@meta` 문법 예약어를 `@config` 나 `@head` 로 업데이트 한다고 가정합시다. 이 텍스트 문서는 당연히 아무런 컴파일 타임 경고 에러를 뱉지도 않고 그저 조용히 방치되어, 수많은 뉴비들에게 에러를 토핑하는 시한폭탄 그 자체가 됩니다. 

**결함 2: 레거시 구 버전 프로젝트명 (Contexta/Graphframe 잔재) 혼재의 치명성**
*   리포지토리가 대규모 리브랜딩되어 타이틀을 `Lynxes`로 갈아입었음에도 불구하고, 수많은 마크다운 문서 구석 예제 블록에 구시대의 산물인 `import contexta` 나 타겟을 상실한 레거시 네이밍 스니펫 변수들이 음습하게 굴러다니고 있을 확률이 높습니다. 의심 없이 복붙을 시전하는 선량한 개발자들은 곧장 터미널에서 빨간 `ImportError`를 겪고 프로젝트 라이브러리 품질 자체에 대한 신뢰를 영구적으로 끊어버립니다.

## 🛠 2. 즉각적인 대공사급 리팩토링 지시 및 아키텍처 규칙

### Rule 1: 마크다운 인라인 '하드코딩' 방식의 완전 철폐 및 `코드 Include(임베드)` 기법 전면 도입 확정
*   **지엄한 수정 지시:** 앞으로 공식 문서 `.md` 블록 내부에 본인이 직접 사람 손으로 코드 로직을 줄줄이 타건하지 마십시오. 금지합니다.
*   가장 먼저 할 일은, 이 Lynxes 깃허브 리포지토리의 최상단 루트 `examples/` 혹은 `tests/` 폴더 내부에, **모든 GitHub Actions (CI/CD) 자동화 파이프라인에서 매번 Push가 일어날 때마다 강제로 컴파일되고 빌드 테스트되어 에러가 없음을 증명받는 "실제 구동용 소스코드 파일"(`.rs`, `.py`)**을 완벽하게 생성해두는 것입니다.
*   그 후 문서를 작성할 때에는 마크다운 매크로 플러그인(`--8<--` 계열 등의 Snippet Injector)을 사용하여 그 실제 파일을 물리적으로 문서 화면에 참조(Embed)하게 당겨오거나, 아니면 최소한 하드코딩 코드 바로 상단에 "※해당 전체 소스 파일은 `[examples/python/01_read.py]` 에 생존해있습니다"라는 확실한 링크 이정표를 삽입하십시오.

### Rule 2: 과거 레거시 키워드 색출(Grep 스캔) 및 완전 멸살
*   문서가 저장된 내 모든 디렉토리 로컬 경로에 대해 강력한 정규식(Regex) `grep` 스캔을 3회 이상 단행하여 `contexta`, `Graphframe` 등(소/대문자 조합까지 모조리 포함)의 과거 단어가 들어있는 예제나 설명이 단 한 줄도 남아있지 않게 소탕 및 분쇄하십시오. 오로지 신규 진영인 `import lynxes as lx` 구문 체계 작동만을 문서는 용인해야 합니다.

### Rule 3: 튜토리얼을 뺨치는 레벨의 극단적이고 병적인 '인라인 주석' 무장 선언
*   예제 소스 파일 자체만 문서에서 거칠게 떼어내 복사하여 메신저 앱으로 다른 개발 팀원에게 무심코 던지더라도, 스크립트 코드 스니펫만 읽고서도 시스템의 전체 아키텍처와 흐름 논리가 파악될 정도로 풍부하게 구성해야 합니다.
*   주석 없이 단순히 `g.collect()` 라고 타건하고 넘어가는 싸구려 예제를 금지합니다.

## 📝 3. (예시) 소스코드 스크립트는 최소한 이 퀄리티로 무장하십시오.

**[실제로 존재하는 `examples/python/02_lazy_expand.py` 파일의 주석 요구 퀄리티 수준]**
```python
import lynxes as lx

# [STEP 1] Data Loading Stage
# 해당 명령은 물리적 파일을 Eager하게 읽어 메모리에 즉각 세팅하지만, 향후 진행될 쿼리 로직은 전적으로 계획 단계(lazy)로 권한을 넘깁니다. 
g = lx.read_gf("examples/data/social.gf")

# [STEP 2] Logical Plan Building 체이닝 파이프라인 형성
# 아래의 filter_nodes 와 expand 명령은 즉시 실행이 아닙니다. 단순히 체이닝되어 AST(초상형 구문트리) 형태로 차곡차곡 논리적으로만 저장되며, 현재 CPU와 I/O 자원을 1kb 조차 소모하지 않고 있습니다.
plan = (
    g.lazy()
    .filter_nodes(lx.col("_id") == "alice")
    .expand(edge_type="KNOWS", hops=2, direction="out")
)

# [STEP 3] Physical Execution 발동 기폭제
# .collect()를 타건하는 바로 이 순간 시스템이 깨어나며 러스트 코어의 Rayon 스레드 풀이 기동, 내부 Optimizer 개입 하에 CSR 고속 이웃 탐색이 폭발적으로 일어납니다.
# 결과 반환물은 Arrow RecordBatch를 소유한 기존과 전혀 엮이지 않은 고속의 독립적 GraphFrame 덩어리입니다.
result = plan.collect()
```
이런 압도적인 설명과 주석이 내포된 파일이 깃허브 C/I 빌드상에서 에러율 0%로 정상 컴파일되는지를 확보부터 한 뒤에야, 문서를 대중에게 작성 노출할 자격이 주어집니다.
