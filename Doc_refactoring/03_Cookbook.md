# 🧨 [Cookbook] 쿡북 문서 규칙 및 작성 가이드 (초강도/상세본)

**"원리나 설계 철학 같은 사담은 빼라. 당장 서비스 장애를 막아야 하는 실무 개발자를 위한 '복붙 가능한 전술 교범'이다."**

현재 Lynxes 문서는 다양한 환경의 장애 상황이나 엣지 케이스 대응 팁들이 제대로 체계화되지 않았으며, `guides/` 와 `cookbook/` 의 경계 또한 모호합니다. 특수한 목적 달성이나 비정형 데이터 처리법 등은 철저하게 '목적 달성 중심(Problem-oriented)'의 쿡북 문서로 적재되어야 합니다.

## 🚨 1. 현재 문서 생태계의 치명적 결함 및 해부

현재 `docs/index.md` 23~27번 라인을 보면 `cookbook/ego-network.md`, `cookbook/shortest-path.md` 등의 쿡북 연결 링크가 존재합니다. 구조적 아이디어는 좋으나, 파일 개별로 내부에 들어갈 콘텐츠의 엄격한 규격(Format)이 정립되어 있지 않습니다.
뿐만 아니라 파이썬 엔진과 무관하게 '대용량 파일 읽기 중 OOM 방지법', '`.gf` 내의 특수 문자열 이스케이프 파싱' 등의 엣지 케이스 처리법들이 메인 튜토리얼 바닥에 기생하며 섞여 있다면 당장 모조리 적출해 독립된 쿡북 문서로 파내야만 검색이 용이해집니다.

## 🛠 2. 즉각적인 리팩토링 지시 및 작성 규칙

### Rule 1: 철저한 '통증(Pain)' 기반의 능동형 제목 짓기
쿡북의 제목은 절대로 라이브러리의 기능 이름(명사형)으로 점철된 추상적인 제목이면 안 됩니다. 실무진이 키보드에 구글 검색으로 다급히 입력하는 '날 것의 질문 문장' 형태여야 합니다.
*   ❌ 나쁜 제목 형태 예시: `Parquet Arrow Interoperability (Parquet Arrow 상호운용성)` - 제목만 보고 무엇을 해결하는지 감이 안 옴.
*   ✅ 강제하는 제목 형태 지침: `ARROW 통신/변환 중 메모리 누수(OOM) 방어하며 대용량 Parquet 그래프 로드하는 방법`

### Rule 2: 쿡북 내부의 '단일 템플릿(4단 콤보)' 절대 준수 의무
앞으로 추가되는 모든 쿡북 하부 문서(`.md`) 파일들은 개인 재량으로 쓰지 마십시오. 예외 없이 치명적으로 아래의 4단계 목차 블록 구조를 강제로 띄워야 합니다.

1.  **[상황 (The Problem)]**: 도대체 "어떤 운영/개발 상황에서 이 코드를 당장 써야 하는가?"를 2줄 이내로 핵심만 요약합니다.
2.  **[환경과 전제 조건 (Prerequisites)]**: 버전 파편화를 막기 위해 `lynxes-core v0.3.0 이상`, `Rayon 활성화` 등의 필수 구동 환경 조건을 경고(Alert) 형태의 블록 박스로 가장 강조하여 선언합니다.
3.  **[해결 코드스니펫 (The Recipe)]**: 불필요한 Print문이나 잔가지가 모두 쳐진, 당장 에디터로 복사-붙여넣기 해도 100% 구동하는 단 하나의 함수. **※사용자 자신의 환경 스펙에 맞추어 수동으로 타건/변경해야 할 변수**는 `[YOUR_MASSIVE_DATA_PATH.csv]` 처럼 눈에 거슬릴 정도의 대문자 브래킷 플레이스홀더로 강제 시각화하십시오.
4.  **[부작용 및 권장 안함 (Side-effects & Anti-pattern)]**: 은탄환은 없습니다. 이 마법의 코드를 남발했을 때 무슨 기술적 부채나 리스크(예: File IO 병목 초래, Rayon 스레드 풀 독점에 의한 타 프로세스 블로킹)가 유발되는지 솔직하게 뒷감당 요령을 기재하십시오.

### Rule 3: 환경 변수와 의존 모듈의 완전 고립주의
*   쿡북에 사용되는 복붙용 코드는, 독자가 문서를 앞에서부터 보지 않고 이 페이지에 처음 구글 검색으로 떨어졌더라도 무조건 구동되어야 합니다.
*   단순한 코드 블록이라고 해서 최상단에 `import lynxes as lx` 같은 기초 모듈 로딩 임포트 구문을 생략하는 만행을 저지르지 마십시오. 완벽히 자립 가능한(Isolated) 온전한 단일 스크립트 파일을 제공하십시오.

## 📝 3. (예시) 이런 템플릿 형태로 찍어내십시오.

**[단일 책임 쿡북 템플릿 적용 모범답안]**

```markdown
## 대용량 `.gf` 파일 OOM 피하며 청크 단위로 분할 읽기 가이드

> [!WARNING]
> ※ 사전 조건: 이 레시피 코드는 `lynxes` >= 0.4.0 버전 환경에서만 유효하며, 로컬 환경에서 최소 16GB 이상의 가용 RAM 공간을 권장합니다.

**1. 발생 문제 상황**: 
1000만 달러 단위, 혹은 수천만 개의 Node/Edge가 선언된 무식한 크기의 레거시 `.gf` 파일을 로드하려 할 때, 러스트 코어 레벨 버퍼에서 `Out of Memory (OOM)` 시스템 패닉이 발생하여 앱이 다운타임을 겪을 때 사용합니다.

**2. 즉효 해결 코드스니펫**:
\`\`\`python
import lynxes as lx

# 주의!!! chunk_size 내부의 인자값은 서비스 시스템의 남은 가용 인메모리 사이즈에 맞추어 직접 수치 조절(Tuning)하세요.
FILE_PATH = "[YOUR_MASSIVE_DATA_FILE_PATH_HERE].gf"

# Eager Load 가 아닌 Lazy File 스트리밍 리더로 접근합니다.
lazy_frame = lx.read_gf_lazy(
    FILE_PATH,
    chunk_size=1_000_000,   # 정확히 백만 행(Row) 단위로 Arrow 버퍼를 안전하게 자릅니다
    parallel=True           # Rayon 코어 병렬 데이터 I/O 스레드를 최대 속도로 개방합니다
)
result = lazy_frame.collect()
\`\`\`

**3. 예상 에러 및 유의사항 (Troubleshooting)**: 
강제로 활성화한 저 `parallel=True` 인자 때문에, 만약 FastAPI나 여타 Tokio 비동기(Async) 런타임 하위 루프 스레드 위에서 이 스니펫을 아무런 래핑 없이 통째로 돌린다면, Rayon과 ThreadPool이 맞물려 데드락(Deadlock)이 터져버릴 수 있습니다. 워커의 메인 비동기 루프와 격리를 철저히 점검하세요.
```
